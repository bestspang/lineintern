# 🛡️ AI Agent Guardrails — Read Before You Edit

> **Audience:** Any AI agent (Lovable, Claude, GPT, Gemini, etc.) about to modify this codebase.
> **Why:** Past AI rounds have repeatedly broken working features by "improving" them. This document is the contract that prevents that.
> **Last updated:** 2026-05-02

---

## 🚨 The 5 Rules That Must Never Be Broken

1. **Functions/files marked `// ⚠️ VERIFIED YYYY-MM-DD` are frozen.**
   Do not refactor, rename, or "clean up" their logic. Only change them when the user explicitly asks for that exact change. If you must touch one, bump the date and document the change in the comment.

2. **No re-adding deleted features.**
   `receipt`, `deposit`, `daily_deposits`, `receipt_quota`, `receipt_approvers` were intentionally removed (Phase 2–4). The `scripts/smoke-test.mjs` enforces this — don't suppress the check.

3. **Never edit migrations after they ship.**
   Each change is a *new* timestamped migration. Editing existing files in `supabase/migrations/` desyncs production from history.

4. **Never edit autogenerated files.**
   - `src/integrations/supabase/client.ts`
   - `src/integrations/supabase/types.ts`
   - `.env`
   - `supabase/config.toml` (only function-specific blocks may be added)

5. **Never sync the LINE webhook URL to `.env`.**
   `.env` and `config.toml` reference the *preview* project ref `phhxdgaiwgaiuecvfjgj`. The LINE Console webhook URL must stay on **LIVE** ref `bjzzqfzgnslefqhnsmla`. Mismatch is expected. See `CRITICAL_FILES.md` rule #7.

## ✅ Pre-Edit Checklist (run mentally before touching code)

For **any** non-trivial change, walk through this in order. Skipping a step causes regressions.

### Step 1 — Identify the surface area
- [ ] Which files am I about to modify?
- [ ] Are any of them in `.lovable/CRITICAL_FILES.md`?
- [ ] Do any contain `// ⚠️ VERIFIED`? If yes, stop and confirm with the user.

### Step 2 — Cross-surface impact
A "feature" usually lives in 4–6 places. Before changing one, check the others are still consistent.

| If you change… | Also verify… |
|---|---|
| A bot command in `command-parser.ts` | (a) Handler exists in `line-webhook/index.ts`, (b) Row exists in `bot_commands` DB, (c) `/help` flex message renders it, (d) `registry-snapshot.json` has it |
| A Portal page in `src/pages/portal/` | (a) Route in `App.tsx` under `<Route path="/portal/*">`, (b) Nav entry in `PortalLayout.tsx`, (c) Action in `src/lib/portal-actions.ts`, (d) Snapshot |
| An Admin page in `src/pages/` | (a) Route in `App.tsx`, (b) Sidebar entry in `DashboardLayout.tsx`, (c) `webapp_page_config` row, (d) Snapshot, (e) `usePageAccess` rule |
| A point-earning rule | (a) `point_rules` DB row, (b) Calculator function (`point-attendance-calculator` etc.), (c) Notification text in `point-monthly-summary`, (d) Help.tsx FAQ |
| Employee schema | (a) Migration adds column, (b) `is_active` ↔ `status` sync trigger, (c) Frontend types regenerate, (d) `EmployeeSettings` form |
| LINE webhook routing | (a) `command-parser.ts`, (b) `command-routing-logic` memory, (c) `bot_commands.available_in_dm` / `available_in_group` flags |

### Step 3 — Verify before claiming done
- [ ] Run `node scripts/consistency-audit.mjs` — must show 0 fail
- [ ] Run `node scripts/smoke-test.mjs --skip-build` — must show 0 fail
- [ ] If you touched UI, open the page in the browser tool and click the affected button

## 🔥 Common Regression Traps (real incidents)

These have all happened before. They are why this file exists.

| Trap | What broke | Prevention |
|---|---|---|
| AI changed LINE webhook URL to match `.env` | All checkout flows died for a day (2026-04-30) | Rule #5 above |
| AI added new commandType without handler in webhook | Command silently no-op'd | Always add to BOTH `command-parser.ts` and `line-webhook/index.ts` switch |
| AI "simplified" `formatBangkokTime` and double-converted | All timestamps shifted +7h | Never touch `_shared/timezone.ts` or `src/lib/timezone.ts` without testing both layers |
| AI removed a portal nav item to "clean up" | Users lost access to Leave Balance | Bottom nav has exactly 6 items — any change requires user approval |
| AI rewrote `Help.tsx` static FAQs from scratch | Lost translations; mismatch with DB | Treat `STATIC_FAQS_TH/EN` as fallback only — DB (`portal_faqs`) is source of truth, only patch missing items |
| AI ran `DROP TABLE` in migration | Lost production data | Always prefer additive: ADD COLUMN, soft-delete with `is_active = false` |
| AI assumed `employees.nickname` exists | TypeScript build broke across 12 files | Schema rule: `full_name` only. Check memory:infrastructure/employee-schema-constraints |

## 🎯 If The User Says "Fix Everything" or "Audit The Whole App"

This is the biggest regression risk. **Do not** start editing. Instead:

1. Run `node scripts/consistency-audit.mjs` and `node scripts/smoke-test.mjs --skip-build`
2. Report what's actually broken vs. what's working
3. Propose a **scoped** plan via `plan--create` — never "rewrite everything"
4. Ask the user which subset to address
5. Tackle one domain at a time (commands, points, attendance, etc.)

The user's exact pain point is *"AI keeps breaking working features when fixing other things."* Honoring that is more important than completing every line of an ambitious plan.
